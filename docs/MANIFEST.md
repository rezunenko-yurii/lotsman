# Lotsman — манифест

## Проблема

ИИ-агент на большой кодовой базе тратит основную часть токенов не на генерацию кода,
а на **навигацию**: поиск нужных мест, чтение файлов целиком, повторное открытие того,
что уже находил. Четыре статьи расхода:

1. **Поиск** — «где это лежит» (перебор grep'ов, обход дерева).
2. **Чтение** — файлы загружаются целиком ради пары строк.
3. **Раздувание контекста** — прочитанное остаётся в окне и оплачивается на каждом шаге.
4. **Блуждание** — агент заново ищет то, что уже видел.

## Решение

**Lotsman** — локальный индексатор кодовой базы с CLI-интерфейсом, спроектированный
как «орган чувств» для ИИ-агента. Принцип: **амортизация** — дорогое вычисляется один
раз (индекс), дешёвое спрашивается много раз (запросы < 1 сек, ответы — десятки строк,
а не тысячи).

Агент вместо чтения файлов вызывает:

| Команда | Отвечает на вопрос | Заменяет |
|---|---|---|
| `lotsman map` | «Как устроен проект? Что важно?» | чтение десятков файлов |
| `lotsman search <q>` | «Где код, отвечающий за X?» | серию grep + чтений |
| `lotsman outline <file>` | «Что внутри файла?» | чтение файла целиком |
| `lotsman refs <name>` | «Кто использует символ X?» | grep по всему репо + чтения |
| `lotsman defs <name>` | «Где определён X?» | grep + чтения |

## Алгоритмическое ядро

1. **Инкрементальный индекс** (sha256 + mtime): при повторном запуске переиндексируются
   только изменённые файлы. Хранилище — SQLite (zero-dependency).
2. **Извлечение символов через tree-sitter**: точные определения (функции, классы,
   методы, типы) с сигнатурами и номерами строк. Regex-fallback для языков без грамматики.
3. **Лексические ссылки**: подсчёт употреблений идентификаторов по файлам —
   языконезависимо, устойчиво к неполному парсингу.
4. **Граф ссылок + персонализированный PageRank** (как в Aider RepoMap): файлы — узлы,
   «файл A употребляет имя, определённое в B» — рёбра. PageRank ранжирует важность,
   персонализация смещает ранги к файлам/идентификаторам в фокусе текущей задачи.
5. **Repo Map под токен-бюджет**: жадный отбор самых важных определений, пока карта
   не заполнит заданный бюджет (по умолчанию 2048 токенов). Агент получает «скелет»
   проекта фиксированной стоимости.
6. **BM25-поиск по символам** с разбиением camelCase/snake_case на подтокены:
   семантически осмысленный поиск без эмбеддингов и API-ключей.

## Принципы проектирования

- **Zero-infrastructure**: stdlib + tree-sitter. Без векторных БД, API-ключей, демонов.
- **Скорость запроса важнее скорости индексации**: индекс — амортизированная разовая
  плата, запросы должны быть мгновенными.
- **Вывод — для LLM**: компактный, детерминированный, с `path:line` для перехода;
  `--json` для машинного потребления.
- **Деградация вместо отказа**: нет tree-sitter грамматики → regex; нет git → обход fs.
- **Инкрементальность везде**: повторная индексация большого репо после правки
  пары файлов — доли секунды.

## Не входит в scope (v1)

- Эмбеддинги / векторный поиск (заготовлен интерфейс, добавляется как слой).
- RAPTOR / GraphRAG-саммари (требуют LLM-вызовов при индексации).
- Межрепозиторный индекс, LSP-сервер, демон с file-watcher.

## v2 — точность и семантика

v1 доказал работоспособность конвейера, но оставил два структурных ограничения:

1. **Лексические ссылки шумят**: параметр `request` неотличим от вызова `request()`.
   v2 переходит на **tags-запросы tree-sitter** (`@reference.call`, `@reference.type` и
   т.п.) — считаются только настоящие семантические ссылки; лексический подсчёт
   остаётся фоллбэком для языков без tags-грамматики.
2. **BM25 ищет по словам, а не по смыслу**: запрос «auth» не найдёт `login_user`.
   v2 добавляет **локальные статические эмбеддинги (model2vec)** — без torch, без
   API-ключей, модель ~30 МБ, эмбеддинг тысяч символов в секунду. Поиск становится
   **гибридным**: BM25 + косинусная близость, слияние рангов через **Reciprocal
   Rank Fusion** (RRF, k=60). Если модель недоступна — деградация до чистого BM25.

Векторы хранятся в той же SQLite-схеме (BLOB в `symbols`), инкрементальность
сохраняется: пере-эмбеддятся только символы изменённых файлов.

## v3 — интеграция с агентом

v2 дал точность; v3 убирает последнее трение между агентом и индексом:

1. **Кэш ранжирования**: PageRank пересчитывался на каждый `map` (~1.2 с на Django).
   Дефолтное ранжирование (без `--focus`/`--mention`) детерминировано для данного
   состояния индекса → кэшируется в SQLite и инвалидируется по слепку состояния
   файлов. Дефолтная карта становится мгновенной; персонализированная считается
   как раньше.
2. **MCP-сервер**: агенту не нужно строить shell-команды и парсить stdout —
   `lotsman mcp` поднимает stdio-сервер (JSON-RPC, протокол MCP), инструменты
   `map / search / outline / defs / refs` с типизированными схемами. Реализован
   на stdlib (~200 строк), без SDK-зависимостей — в духе zero-infrastructure.
3. **Боевое подключение**: политика навигации внедряется в CLAUDE.md реального
   проекта + `.mcp.json` для автозапуска сервера; сценарий «задача агента»
   прогоняется на Django-индексе с оценкой экономии токенов.

## Критерии успеха

- Индексация репозитория ~1000 файлов — секунды; повторная без изменений — < 0.5 с.
- `map/search/outline/refs` — < 1 с на среднем репо.
- Repo map укладывается в бюджет и содержит действительно центральные символы.
- Агент, следуя README-политике, отвечает на «где/кто/как устроено» без чтения файлов.
